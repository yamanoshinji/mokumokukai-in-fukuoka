### 00:10-00:18 基本概念（プロンプトは「お願い文」ではなく「仕様」）

#### 進め方: 「誤解」→「原則」→「実装上の型」→「今日の作業」の順

#### ステップ1: よくある誤解（1-2分）

話すこと:

```code
初心者が最初にやりがちなのは、
「いい感じに答えてください」を長文化して祈ることです。

でも実際は、LLMは仕様が曖昧だと曖昧に返します。
「丁寧に」「簡潔に」「箇条書きで」「100文字以内」など
制約を具体化しないと、品質は安定しません。
```

板書: `曖昧な依頼 = 曖昧な出力`

---

#### ステップ2: プロンプト設計の3層（2-3分）

話すこと:

```code
プロンプトは最低3層で考えると整理しやすいです。

1. 役割（Role）
   例: 「あなたは丁寧な日本語アシスタント」

2. 制約（Constraint）
   例: 「不明な事実は推測しない」「3点以内で回答」

3. 出力形式（Format）
   例: 「JSONで返す」「見出し+箇条書き」

この3つを分離して管理すると、
どこを変えたら何が変わったかを追跡できます。
```

板書:

```code
[Role] 何者として答えるか
[Constraint] 何を守るか
[Format] どう出力するか
```

---

#### ステップ3: なぜ分離が必要か（2分）

話すこと:

```code
1つの長文プロンプトに全部書くと、
変更時に差分が追えず、品質改善が属人的になります。

だからコードでは、
- system prompt（不変方針）
- user prompt（毎回の入力）
- optional context（追加情報）
に分けます。

この分離が、次回以降のツール呼び出し指示にも直結します。
```

板書: `分離 = 保守性 + 検証性`

---

#### ステップ4: 評価軸を先に決める（2-3分）

話すこと:

```code
改善で一番大事なのは、
「何が良くなったかをどう判定するか」です。

今回は評価軸を3つに固定します。
1. 指示遵守（制約を守れているか）
2. 明瞭性（読みやすいか）
3. 事実性（不明を不明と言えているか）

毎回この3軸でログ評価すれば、
感覚ではなく記録で改善できます。
```

板書:

```code
改善ループ:
プロンプト変更 → 同一質問セットで実行 → 3軸評価 → 次の変更
```

---

```code
第1回: つながるCLI
第2回: 狙って答えさせるCLI（今日）
第3回: 外部ツール呼び出し

つまり今日は、
「エージェントの行動品質を決める設計図」を作る回です。
```

### 00:18-00:25 失敗パターンと対策

話すこと:

- **失敗1：1回で複数要素を変更して原因不明**
  - 問題：何が効いたのか判定不能
  - 対策：**1変更1評価**を徹底
- **失敗2：良し悪しを感覚で判断**
  - 問題：3人の評価が揃わない
  - 対策：**共通評価軸（指示遵守・明瞭性・事実性）**で採点
- **失敗3：制約が曖昧で破られる**
  - 問題：「簡潔に」が人によって違う
  - 対策：**数値化（例: 120文字以内、3項目以内）**

対策のルール（3つ）:

1. プロンプトは役割/制約/形式に分離
2. 評価用の固定質問セットを使う
3. 改善ログ（何を変えたか）を残す

#### 操作2-1: `src/prompt.ts` を新規作成

```typescript
export const PROMPT_VERSION = 'v2.0.0';

export const SYSTEM_PROMPT = `
あなたは日本語で回答する実務アシスタントです。
以下を必ず守ってください:
1. 不明な事実は「不明」と明示し、推測で断定しない
2. 回答は簡潔に、原則120文字以内
3. 箇条書きが適切な場合は最大3点
`;

export function buildUserPrompt(userInput: string): string {
  return `ユーザー質問: ${userInput}`;
}
```

## 4. 詰まりポイントと解決策

### パターン1: 指示を守らない

症状:

```code
- 120文字を超える
- 箇条書きが5点以上出る
```

原因と対策:

| 原因 | 対策 |
| --- | --- |
| 制約が曖昧 | 数値制約を明記（例: 最大3点） |
| 制約が多すぎる | 優先順位を明記（最優先: 事実性 など） |
| 質問が曖昧 | `buildUserPrompt` 側で文脈補足を追加 |

---

### パターン2: 回答が短すぎて情報不足

症状:

```code
一言で終わって実用性がない
```

原因: 文字数制限が厳しすぎる

対策:

```typescript
// 例: 120文字 → 180文字へ緩和
```

---

### パターン3: 改善したはずなのに比較できない

症状:

```code
どの変更の結果かわからない
```

原因: `promptVersion` 未更新、変更記録なし

対策:

```code
- プロンプト変更時は必ず version を更新
- 変更内容を1行メモで残す
```

---

### パターン4: 同じ質問でも評価が割れる

症状:

- Aさんは高評価、Bさんは低評価

原因: 評価基準が言語化されていない

対策:

```code
3軸の採点基準を先に定義
遵守: 制約違反の有無
明瞭: 読みやすさ
事実: 不明時の断定回避
```
