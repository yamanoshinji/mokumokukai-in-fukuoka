## 1. ドキュメントも「コード」と一緒に管理しよう

設計書やDB定義書を、プログラム本体（Gitリポジトリ）と同じ場所で管理しましょう

- 理想の構成： フロントエンド、バックエンド、設計書を一つの箱（モノレポ）にまとめる
- メリット： AIが「このコードの背景にある設計は何か？」を瞬時に参照できるようになり、回答の精度が爆上がりします

[!TIP] 「設計と実体のズレ」をなくす
DB定義はSQLファイル、API定義はOpenAPI（yml）で管理するのがベストです
ドキュメントがリポジトリ外にあると、AIに教える手間が増えてしまいます

## 2. 「構造化」されたドキュメントがAIの好物

AIに設計を理解させるには、 **「構造化（決まったルールで書くこと）」** が重要です

### Markdown（マークダウン）を活用

この投稿のように、見出しや箇条書きのルールが決まっている形式です
AIは「どこが重要か」を正確に把握できます

### Excelで書く場合の工夫

どうしてもExcelが必要な場合は、セルの位置や書き方のルールを厳格に決めましょう
ルールさえあれば、プログラムでAIが読みやすい形式（Markdown等）に自動変換できます
添付の Python ファイルは生成AIに作らせた変換ツールです

## 3. 画面イメージや図は「画像」ではなく「データ」で描く

Excelやペイントで描いた図は、AIにはただの「色の塊」に見えてしまいます

- 推奨ツール： Draw.io（VSCodeで使えます）や Figma
- なぜ？： これらは内部的に「テキストデータ（XML）」でできています
- Drawio の例:
  添付の S001_文書検索画面.drawio
  VSCode の拡張機能（Draw.io integration）を使って開いてみましょう
- 驚きの活用法：
  AIにDraw.ioのファイルを読み込ませると、 **「この図からVue.jsの画面モック（試作品）を作って」** という指示だけで、コードを書き出してくれるレベルに達しています。

## 4. 設計の「ケアレスミス」や「備忘録」はAIに丸投げ！

誤字脱字、番号のズレ、論理的な矛盾のチェックに人間が時間を使うのはもったいないです
レビューの自動化： 自分で書いた設計書を、まずは生成AIにチェックさせましょう
添付の「S101_design_review.md」に生成AIによるレビュー結果の例を示します

### 高度な相談

「この設計変更をしたら、どこに影響が出る？」といった影響範囲の調査も、AIは得意です

ドキュメントの一式、顧客との議事録などを NotebookLM にすべて登録しておけば、過去の会話履歴からの検索も簡単にできます
「この仕様について、いつの議事録で誰が明言していた？」と聞くだけで、すぐに答えが返ってきます

## まとめ：設計にも生成AIを活用

AIは、情報を整理して渡せば渡すほど、強力なパートナーになります
添付のプロンプト（指示書）を使えば、GitHub Copilot上で簡単に試せます
