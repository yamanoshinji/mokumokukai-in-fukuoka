#### 手順4: CLI化（20分）

目的: 固定入力から動的入力へ。複数回の対話を可能に

実装内容:
- `readline` モジュールで標準入力を受け取り
  ```typescript
  import readline from 'readline';
  
  const rl = readline.createInterface({
    input: process.stdin,
    output: process.stdout
  });
  
  const askQuestion = (prompt) => {
    return new Promise(resolve => {
      rl.question(prompt, resolve);
    });
  };
  ```
- ループで複数回の対話を実現
  ```typescript
  while (true) {
    const userInput = await askQuestion('you: ');
    if (userInput.toLowerCase() === 'exit') {
      console.log('👋 終了します');
      rl.close();
      break;
    }
    const response = await llm.call(userInput);
    console.log('assistant:', response);
  }
  ```

完了チェック（すべて✓になったら次へ）:
- [ ] `npm start` を実行後、プロンプト `you: ` が出現
- [ ] 手動で質問を入力 → LLM応答が返される
- [ ] 複数回やり取り可能
- [ ] `exit` を入力→正常に終了

失敗時ガイド:
- プロンプトが出ない: `readline` が正しくセットアップされているか確認

| 原因 | 対策 |

| キー読み込み順序がずれている | `dotenv.config()` が最初の行？ |

最初の確認ステップ:
```typescript
console.log('APIキー:', process.env.OPENAI_API_KEY?.slice(0, 10) + '...');
console.log('使用モデル:', process.env.MODEL_NAME);
```

### パターン2: レスポンス形式を取り違える（`undefined` エラー）

症状:
```
TypeError: Cannot read property 'text' of undefined
```

原因: LLMライブラリのレスポンス構造を誤解

対策:
```typescript
// まず生レスポンスをダンプ
console.log('Raw Response:', JSON.stringify(response, null, 2));

// それから取り出し処理の修正
const text = response.choices?.[0]?.text 
  || response.content?.[0]?.text 
  || response.message?.content;
console.log('Extracted text:', text);
```

### パターン3: 文字化け・改行崩れ

症状:
- ターミナル表示が見づらい
- 日本語が `???` になる

原因: 文字エンコーディング設定

対策:
```typescript
// stdoutを明示的にUTF-8に
console.log(response.replace(/\r\n/g, '\n').trim());

// Node.jsコマンドラインで指定
// node --input-type=module script.ts
```

### パターン4: 例外でプロセスが即座に終了

症状:
- エラーが出ると即座に落ちて、ログが見えない

原因: グローバルな例外ハンドリングなし

対策:
```typescript
process.on('unhandledRejection', (reason, promise) => {
  console.error('❌ キャッチされない非同期エラー:', reason);
  fs.appendFileSync(logFile, JSON.stringify({
    timestamp: new Date().toISOString(),
    type: 'uncaught_rejection',
    error: String(reason)
  }) + '\n');
  process.exit(1);
});
```

### 宿題1: 同じ質問の応答揺れを記録
- 固定プロンプト、固定質問を**3回実行**
- 「temperature: 0.2」と記録
- 応答がどの程度揺らぐか、違いを記録

**記録フォーマット:**
```
【質問】「2+2は何ですか？」
【温度】0.2
実行1: 「2 + 2 = 4 です」
実行2: 「4です」
実行3: 「答え：4」
【観察】意味は同じだが語尾が異なる。数学的誤りなし
```

### 宿題2: プロンプト1箇所の変更実験
- 現在の固定プロンプト を A とする
- **1つの変更だけ加えた** プロンプト B を作る
- 例：「です」→「です 😊」、「答えて」→「日本語で答えて」
- 同じ質問を5回ずつ実行し、応答の変化を記録

**記録フォーマット:**
```
【変更内容】プロンプト末尾に「。簡潔に。」を追加
【変更前（A）】「...と質問されています。返答してください」
【変更後（B）】「...と質問されています。返答してください。簡潔に。」
【結果（5回平均）】
 A: 平均400トークン
 B: 平均150トークン
【察】制約を追加すると文体が大幅に短縮される
```

### 宿題3: 改善案を1つだけメモ化
- 上記1・2の実験から「**これはいいな**」という改善アイデア1つを選ぶ
- なぜそれがいいのかを3行以内でメモ
- 「過剰最適化」は禁止。1つだけ

**記録フォーマット:**
```
【改善案】温度をデフォルト0.2のままで、エラー時に一時的に0.5にリトライする
【理由】
 - 正常系は再現性を保つ
 - 使用額が大きく増えない
 - 「応答がない」という失敗を部分的に救える
```

## 8. ファシリテーター向けメモ

### 進行哲学
- **「速さ」より「共通理解」を優先**
  - 3人同期学習が勉強会の定義。1人が落ちたら全員止まってOK
- **進まないときは機能追加を止め、疎通確認に戻る**
  - ハンズオン途中のアイデア実装は反対。完成まで直進
- **完成したことを必ず口頭化する**
  - 「でき た」→「なぜできたか」→「次に何ができるか」のステップで説得力が増す

### トラブル対応の優先順位
1. **API疎通**：これが通らないと全止め→即セッション時間使って対策
2. **CLI ループ**：途中だと先に進めない→15分で決着つける
3. **ログだけ未完成**：次回に持ち越してOK→「記録の大事さ」は学んだ

### 最後のふりかえり（10分）

声かけの例：
- 「今日API呼び出しで失敗したのは何が原因絡った？」
- 「温度を0.2にすると応答が安定する理由、分かった？」
- 「『今日のコード』を3年後に見返すときに必要な情報、何だと思う？」→「ログです」
