## 1. Playwright で「ブラウザ操作」を資産化する

E2E（エンドツーエンド）テストとは、ユーザーが実際にブラウザを操作する一連の流れをテストすることです

E2Eのツールとしては、一般的によく使われている Playwright の採用を推奨します

https://playwright.dev/

VSCode の拡張機能からも利用でき、Excelマクロのように操作を記録することも可能です

### 自動化のメリット

ログイン、データ登録、検証、後片付けまでをコード（Node.js / Python等）で自動化できます
一度作れば、何度でも同じ条件で実行（冪等性の確保）が可能です

### AIとの相乗効果

AI（Playwright MCP経由）にテストコードを参照させることで、AIは「どう操作すれば期待通りに動くか」を即座に理解します
デバッグをAIに任せる際も、格段に効率が上がります

### クラウド実行

GitHub Actions等で並列実行すれば、ローカル端末の性能に縛られず、大量のテストを短時間で完了できます
下図はフロント、バックの単体テストを並列実行後、E2Eテストを実行している例です
実行結果、実行ログやエビデンスをダウンロードすることもできます

## 2. 「テスト設計」の落とし穴：網羅率100%は正解か？

「最新のAI（Claude 4.6等）を使えば、テストカバレッジ（網羅率）100%も可能では？」と思うかもしれません
しかし、STOとしては**「戦略的なテーラーリング（調整）」**を求めます

### カバレッジ100%を目指さない理由

網羅率を追求しすぎると、設計変更のたびにテストの修正工数が膨れ上がり、費用対効果が悪化します

参考記事: 「テストカバレッジ100%を追求しても品質は高くならない理由と推奨されるカバレッジの目標値について」
https://qiita.com/odekekepeanuts/items/d02eb38e790b93f44728

### フェーズに合わせる

- 開発序盤： 重要なフロー（正常系）に絞って低コストで守る
- 結合試験： 網羅性を高め、エッジケースを固める

### 計画がすべて

「いつ、どこまで、何をテストするか」を人間が先に計画してください
その計画さえあれば、実装自体はAIが正確にこなしてくれます

## 3. リファクタリングこそ「E2Eテスト」の出番

「機能は変えずに、内部コードを綺麗にする」リファクタリングは、保守性を高めるために不可欠です
しかし、テストがない状態でのリファクタリングは、暗闇で刃物を振り回すような危うさがあります

### 安全装置としてのテスト

変更前にE2Eテストを走らせ、変更後も「パス」することを確認できれば、デグレのリスクは最小限になります

### AIによるリファクタリング

1. AIに現状のコードの問題点をレビューさせる
2. リファクタリング計画を立てさせる
3. E2Eテストで動作を担保しながら、AIにコードを修正させる

このサイクルを回すことで、複数人で開発してバラついたコード品質を、安全かつ高速に統一できます

## まとめ：テストは「AIへの信頼」の積み重ね

E2Eテストを整備することは、単なるバグ探しではありません。

- AIが自走できる環境を作る： テストコードはAIにとっての「操作説明書」になります
- コスト意識を持つ： 網羅率に固執せず、プロジェクトの状況に合わせてテスト範囲を「テーラーリング」すること
- 変化を恐れない： 強固なテストがあれば、リファクタリングも設計変更も迅速に対応できます

「テストを書く時間は、未来の時間を買う投資」です
AIにテストを書かせるコツを掴み、変化に強いシステムを構築していきましょう
